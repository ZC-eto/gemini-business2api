# Chat Proxy Rotate Design

**目标**

在不破坏现有 `Resin / 普通 HTTP(S) / SOCKS / 直连` 兼容性的前提下，为聊天链路补齐“代理问题优先切聊天代理、账号问题继续切账号”的判定逻辑，并同步收敛验证码发送状态策略，提升成功率、速度和可观测性。

**范围**

- 聊天链路新增基于异常类型的聊天代理切换策略。
- 仅 Resin 聊天代理支持自动 rotate；普通代理和直连只记录状态与提示。
- 账号管理页常驻状态栏补齐聊天链路最近一次代理切换结果与原因。
- 验证码发送状态调整为：
  - `confirmed`：完整轮询一轮
  - `unknown`：也完整轮询一轮
  - `failed`：明确失败立即切代理

## 一、当前问题

1. 聊天链路当前只会切账号/冷却账号，不会主动切聊天代理；当聊天代理出口被风控时，容易直接出现 `403`。
2. 聊天代理异常类型没有区分“账号问题”和“代理/IP 问题”，容易误切账号或重复打到坏出口。
3. 验证码发送状态存在误差，`unknown` 和 `failed` 容易混淆；同时此前 `unknown` 短窗口过于激进，不符合实际邮件到达延迟。
4. 页面虽然已有运行态状态栏，但聊天链路的切换结果和原因还不够完整，排障仍要翻日志。

## 二、聊天链路切换策略

### 1. 按异常类型分类

#### 账号问题，不切聊天代理

- `429`
  - 继续沿用现有逻辑：冷却账号 / 切账号
  - 原因：更像额度、频率或配额问题
- `401`
  - 先走现有 JWT 刷新
  - 若刷新后仍失败，再按账号问题处理
  - 原因：更像登录态失效，不是出口 IP 问题

#### 代理/IP/链路问题，允许切聊天代理

- `403`
  - 第一次出现：先 rotate 一次聊天代理，并用同账号重试一次
  - 若同账号换代理后仍是 `403`：视为账号问题，交给现有账号处理逻辑
- `Timeout / TLS / httpx 网络错误 / 504`
  - 直接 rotate 聊天代理，并用同账号重试一次
  - 原因：这些更像链路不通、出口被拦或代理质量差

#### 模糊问题，先不改现有逻辑

- `502`
  - 保持当前逻辑：只切账号，不立刻切聊天代理
  - 原因：当前项目里不少 `502` 是空响应、思考模型无最终内容、会话异常等，不一定是代理问题

### 2. 单请求保护

为了避免请求内死循环：

- 单次聊天请求内，聊天代理 rotate 最多 `1 次`
- 同账号因 rotate 的重试最多 `1 次`
- 超过后退回现有切账号/返回错误路径

## 三、三种代理模式兼容

### Resin 聊天代理

- 支持自动 rotate
- 聊天异常命中规则时调用 Resin rotate API
- 状态栏显示：
  - 平台
  - 业务账号
  - 当前出口 IP / 地区 / 线路
  - 最近一次 rotate 结果 / 原因 / 时间

### 普通 HTTP / SOCKS

- 不支持程序内主动 rotate
- 仍显示当前出口 IP / 地区 / 线路 / 最近错误
- 日志明确提示：当前为标准代理，无法自动切换

### 直连

- 不支持 rotate
- 仅显示服务器当前出口 IP / 地区
- 日志明确提示：当前为直连模式，无法切换代理

## 四、验证码发送状态策略

### 1. 发送状态定义

- `confirmed`
  - 明确成功，可完整等待邮件
- `unknown`
  - 状态不确定，但仍允许完整等待一轮
  - 用配置值控制，例如 `30s / 5s / 6 次`
- `failed`
  - 明确失败，立即结束本轮并交给上层 rotate

### 2. 关键约束

- `failed` 只保留给真正明确的失败证据
  - 明确网络失败
  - 明确的发送失败提示
  - 明确的上游失败信号
- “进入验证码页但未找到重发按钮”不能直接判 `failed`
  - 应视为 `unknown`

## 五、日志与状态栏

### 1. 聊天日志新增

- `♻️ Chat Resin rotate accepted: chat/account_001`
- `⚠️ 聊天请求命中代理切换条件: HTTP 403`
- `⚠️ 当前聊天代理为标准代理，无法自动 rotate`
- `⚠️ 当前聊天链路为直连模式，跳过 rotate`
- `❌ 聊天代理 rotate 后同账号仍返回 403，改按账号问题处理`

### 2. 账号页状态栏补充

聊天链路状态卡补齐：

- 当前模式
- 当前出口 IP / 地区 / 线路
- 最近一次 rotate 结果
- 最近一次 rotate 原因
- 最近更新时间
- 当前是否冷却

## 六、预期收益

### 好处

- 当聊天代理出口被风控时，可以更快换到新出口，而不是直接把账号打入失败路径。
- `403` 会先给代理一次机会，减少误伤账号。
- 页面可以直接看到聊天链路有没有真的切代理、切完是否成功。
- 验证码 `unknown` 保持完整轮询一轮，更符合邮件到达存在延迟的现实情况。

### 风险

- 某些 `403` 可能本质是账号问题，先切代理会增加一次额外重试成本。
- 普通代理和直连没有自动 rotate 能力，只能观测与提示。
- 聊天请求内加入“同账号换代理再试一次”后，异常分支会更复杂，需要小心避免重复记账、重复冷却或重复创建会话。

## 七、测试重点

- Resin 聊天代理：
  - `403` 首次是否 rotate
  - rotate 后同账号仍 `403` 是否回退为账号问题
  - `Timeout / TLS / 504 / httpx` 是否触发聊天 rotate
- 普通 HTTP / SOCKS：
  - 是否只记录状态、不误调 Resin rotate
- 直连：
  - 是否只记录直连出口并提示不可 rotate
- 验证码发送状态：
  - `confirmed` 与 `unknown` 是否都完整轮询一轮
  - `failed` 是否立即返回上层切换 auth 代理
