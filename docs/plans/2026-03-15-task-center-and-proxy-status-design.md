# Task Center And Proxy Status Design

## 背景

当前 `gemini-business2api` 已经支持：

- 账户操作代理 / 聊天操作代理两条主要代理配置；
- 账户注册、刷新、聊天链路的运行态代理记录；
- 任务管理、定时任务、历史记录；
- 设置页手动代理测试；
- GitHub Actions 构建镜像并通过 Dokploy API 更新部署。

但现状存在四个明显问题：

1. 任务管理仍然挂在 `Accounts.vue` 的弹窗里，内容已经超出弹窗适配范围。
2. 代理状态对外展示成了三张卡（`auth` / `mail` / `chat`），和用户真实配置模型不一致。
3. 设置页保存代理后不会立刻得到“当前出口 IP / 地区”的结果，要等真实流量跑起来才会刷新。
4. GitHub Actions 的 `push` 自动触发在当前 fork 仓库上没有生效，虽然 `workflow_dispatch -> GHCR -> Dokploy` 已验证可用。

## 已确认约束

- 对外主展示只保留两条链路：
  - `账户操作代理`
  - `聊天操作代理`
- `临时邮箱代理` 不作为独立主卡展示；若 `mail_proxy_enabled=true`，则视为“临时邮箱轮询复用账户代理”。
- 保存设置后自动跑一次 **HTTP 代理测试**，用测试结果直接刷新运行态显示。
- 浏览器测试保留手动按钮，不在保存设置时自动触发。
- 任务管理需要改成独立页面，不再使用弹窗承载主流程。
- 当前阶段先不额外扩展普通 `HTTP/SOCKS` 的复杂轮换能力，只保证现有显示与逻辑正确。

## GitHub Actions 触发问题结论

已验证事实：

- 仓库存在真实 `PushEvent`，例如 `main` 的 `8ae4c5ccadf13e28751dd89e086babd4ee00f452`。
- 仓库 `Build And Deploy Image` workflow 处于 `active` 状态。
- `workflow_dispatch` 可以成功构建 GHCR 镜像并自动更新 Dokploy。
- `actions/runs?event=push` 返回长期为 `0`，说明 **GitHub 并未为这些 push 生成 workflow run**。

因此可确认：

- 问题不在当前 workflow YAML 的 `on.push.branches` 写法；
- 问题也不在 Dokploy 部署步骤；
- 当前症状更接近 **GitHub 对这个 fork 仓库未实际派发 push workflow 事件**。

本轮实现不试图在应用代码里“修复 GitHub 平台行为”，而是：

1. 保留现有 workflow；
2. 在这轮代码 push 后再次验证是否仍无 `push` run；
3. 若仍无自动触发，则将问题记录为仓库/平台级异常，并给出后续替代路径建议（独立仓库、脱离 fork 网络、或外部触发器）。

## 设计目标

### 1. 代理状态展示收敛

设置页与账号页顶部都只展示两张主卡：

- `账户操作代理`
- `聊天操作代理`

每张卡包含：

- 当前模式：`代理中 / 直连 / 未使用`
- 当前代理类型：`Resin / 普通代理 / 自动检测`
- 最近一次测试或真实流量使用到的出口 IP
- 地区 / 线路 / 延迟
- Resin 平台/账号（若适用）
- 最近一次切换结果与原因（若适用）
- 更新时间

账户卡额外展示：

- `临时邮箱轮询：复用账户代理` 或 `临时邮箱轮询：直连`

这能保留邮箱是否跟随账户代理的事实，同时避免把“邮箱轮询”误理解成第三条独立配置代理。

### 2. 保存设置后的即时状态刷新

设置页点击“保存设置”后：

1. 先保存后端配置；
2. 如果 `proxy_for_auth` 非空，则自动跑一轮 HTTP 测试；
3. 如果 `proxy_for_chat` 非空，则自动跑一轮 HTTP 测试；
4. 将测试结果写入运行态缓存；
5. 重新刷新设置页与账号页顶部卡片。

这样无需等待后续真实流量，页面立刻能看到当前配置对应的出口 IP / 地区。

### 3. 任务管理改为独立页面

新增独立页面，例如 `/tasks`，承载：

- 当前任务
- 定时任务
- 历史记录
- 任务日志
- 顶部固定代理状态区（账户 / 聊天两张卡）

账号页中的 `任务管理` 按钮改成跳转，而不是打开弹窗。

当前弹窗中的主要逻辑与交互保留，但搬迁到独立页面组件中，避免 `Accounts.vue` 持续膨胀。

## 数据流设计

### 运行态来源

运行态仍基于现有 `proxy_runtime` 结构，但对前端展示做聚合：

- `auth`：主展示卡片
- `chat`：主展示卡片
- `mail`：仅作为 auth 卡附属信息来源，不单独成卡

### 设置页保存后的测试写入

新增或扩展后端辅助逻辑，使 `proxy-test` 的结果可以被写入运行态缓存。

这样：

- 设置页自动测试会刷新运行态；
- 账号页/任务页因为共用同一个运行态接口，也能同步看到最新出口信息。

## 错误处理

- 代理保存成功但测试失败：
  - 设置仍保存；
  - 页面提示“保存成功，但代理测试失败”；
  - 状态卡显示最近错误。
- 代理为空：
  - 不自动测试；
  - 状态卡回到 `未使用` 或 `直连`。
- 任务页打开时无任何任务：
  - 顶部仍显示代理状态卡；
  - 内容区显示空状态，不影响查看历史。

## 测试策略

本轮至少执行：

- Python 语法检查：
  - `python -m py_compile main.py core/proxy_utils.py core/register_service.py core/login_service.py core/gemini_automation.py`
- 前端最小类型/构建检查（若本地依赖已具备）；
- 代理状态保存后自动测试的最小接口级冒烟；
- 乱码扫描：
  - 检查新增/修改文件是否出现常见 mojibake 特征。

## 风险

1. 将任务弹窗拆成独立页面时，若直接复制逻辑，容易造成 `Accounts.vue` 与新页面状态分叉。
2. 保存设置后自动测试会额外增加两次代理探测请求，但这是可接受成本。
3. GitHub `push` 不自动触发 workflow 的根因大概率不在应用代码层，可能无法在本轮彻底“代码修复”。
