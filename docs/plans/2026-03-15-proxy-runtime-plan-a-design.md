# Proxy Runtime Plan A Design

**目标**

在不破坏现有 `Resin / 普通 HTTP(S) / SOCKS / 直连` 兼容性的前提下，提升注册/刷新链路的成功率与速度，并在账号管理页常驻显示当前任务实际使用的代理出口信息。

**范围**

- 账号管理页新增常驻代理状态栏，不再依赖弹窗查看。
- 验证码发送状态从“只记录”升级为“驱动切换策略”。
- 增加轻量级坏链路冷却池，避免短时间反复复用明显失败的出口。
- 保持现有 Resin rotate、普通代理和直连模式兼容。

## 一、当前问题

1. `send status=failed` 时仍然会继续邮箱轮询和页面内重发，导致单轮失败耗时过长。
2. 运行中只能通过日志倒推当前出口，排障成本高。
3. 明显失败的出口没有冷却机制，可能在短时间内反复复用。
4. `unknown` 状态没有被区别对待，既可能误等，也可能误切。

## 二、方案 A

### 1. 常驻代理状态栏

位置：账号管理页顶部工具栏下方，固定显示 3 张状态卡：

- 账户操作链路
- 临时邮箱链路
- 聊天链路

每张卡显示：

- 当前模式：`Resin / HTTP / SOCKS / 直连 / 未使用`
- 当前出口 IP
- 当前地区
- 线路/组织
- 业务账号或邮箱
- Resin 平台 / Resin 账号（若适用）
- 最近一次切换结果：`未切换 / 已切换 / 切换失败`
- 最近一次错误
- 最近更新时间

说明：这些状态展示的是“最近一次真实流量”而不是静态配置。

### 2. 验证码发送状态策略

- `confirmed`：进入邮箱轮询。
- `failed`：立即结束当前轮次，直接交给上层 rotate/重试。
- `unknown`：
  - 首次允许一次短窗口轮询；
  - 若仍未收到验证码，则结束当前轮次并切换代理；
  - 不再走长时间轮询 + 多轮页面内重发。

### 3. 坏链路冷却池

增加应用内轻量运行态冷却池，按“最近一次真实出口”记录失败结果。

纳入冷却的错误：

- `send code failed`
- `captcha_check_failed`
- 验证码轮询超时且发送状态非 `confirmed`
- `403`

冷却行为：

- 命中失败后记录失败次数和到期时间。
- 再次准备使用相同出口时，如果仍在冷却期：
  - Resin：优先再发一次 rotate，争取换到别的出口；
  - 普通代理：只记录状态并提示，不能主动切换；
  - 直连：只记录状态，不做自动切换。

### 4. 三类代理兼容

#### Resin

- 使用现有模板解析与 rotate API。
- 状态栏显示平台 / 账号 / 当前出口。
- 失败时优先 rotate。

#### 普通 HTTP / SOCKS

- 继续使用当前代理格式。
- 状态栏显示代理类型、掩码后代理地址、当前出口。
- 冷却池只做“坏链路提示”，不做自动 rotate。

#### 直连

- 状态栏显示服务器当前出口 IP 与地区。
- 失败时不做自动切换，只保留告警信息。

## 三、预期收益

### 好处

- 显著减少“明确失败后仍等待 25-30 秒”的空耗。
- 更快淘汰明显被风控的出口。
- 排障时可以直接看到当前任务到底在用哪个出口。
- 与现有代理配置兼容，不要求用户改代理格式。

### 风险

- `unknown` 提前切换可能误杀少量本可成功的链路。
- 冷却池只在当前服务实例内生效，重启后会丢失。
- 普通代理和直连没有 Resin 那样的主动切换能力，只能展示和提示。

## 四、测试重点

- Resin 模式下：
  - `failed` 是否立即结束本轮并 rotate。
  - 状态栏是否正确显示 Resin 平台、账号、出口。
- 普通 HTTP / SOCKS：
  - 状态栏是否显示出口 IP / 地区。
  - 失败时不会误触发 Resin rotate。
- 直连：
  - 状态栏是否显示服务器当前出口 IP / 地区。
- 任务运行中：
  - 状态栏是否持续刷新。
  - 日志行为与状态栏是否一致。
